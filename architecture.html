The Big Picture

One sentence, then the map

A shopper uses a website. A tiny script riding along quietly notices what they do and mails those notes to a Go API. The API runs every note through a reusable pipeline library that cleans, filters and enriches it, then stores the result so a dashboard can show live charts.

Piece 1 of 4 · The Front Door

🛍️ The Website (e-commerce SPA)

This is the shop the user actually clicks around in. It runs entirely in the browser as a Single Page Application.

What the user can do

Sign up / Log in Browse products Search & filter Add to cart Multi-step checkout See live dashboard

Products come from a free public API (DummyJSON). The cart is saved in the browser so it survives a refresh or a login redirect.

Two ways it talks to the API

Login / data
Calls the Go API with credentials: "include" so the secure cookie travels along.
The JWT
Lives only in an HttpOnly cookie — JavaScript can never read it (safe from XSS).
Tracking
Loads tracker.js via a <script> tag, exactly like a real third-party site would.
Charts
The dashboard reads GET /api/stats to draw live numbers.

Key idea: tracking is invisible & non-blocking

The shop never waits on tracking. If the API is down, the page still works perfectly — the script just quietly stores events for later. A consent banner gates everything: nothing is sent until the user accepts.

Piece 2 of 4 · The Quiet Observer

📡 The Tracking Script (tracker.js)

A small standalone file you drop onto any site. It exposes window.mable and turns user actions into well-formed events for the API. Its golden rule: never break or slow down the host page.

What it captures (in the browser)

PageViewClick AddToCartCheckout PaymentInfoAddedPurchase Lead (email form)

Plus context the browser can see: user agent, language, timezone, screen size, referrer, URL, session id, and any logged-in user id.

What it deliberately does NOT do

It never figures out your IP address or geo-location. Why? You can't trust the client for that. The server adds IP, receipt time and geo. This is the famous browser-vs-server split.

The clever bits that keep it bulletproof

🪶 Non-blocking transport

Uses navigator.sendBeacon (fire-and-forget, survives page close), falling back to fetch(keepalive). Events are batched and flushed every few seconds or when the tab hides.

🔁 Resilient retry

If a send fails, events are saved to localStorage and replayed automatically when the browser comes back online.

🔒 Privacy-aware

Nothing leaves the browser until setConsent(true). No consent → events wait in a local queue.

📐 Contract-correct

Every properties value is coerced to a string to match the API's map[string]string. amount & currency are promoted to top-level fields.

The data layer (no events lost before load)

<script src="/tracker.js"></script>
<script>
  window.mable.init({ endpoint: "http://localhost:8080/api/events" });
  window.mable.setConsent(true);
  window.mable.addToCart({ id: 42, amount: 19.99, currency: "usd" });
</script>

A Google-style window.mableDataLayer queue means calls made before the script finishes loading are kept and replayed.

Piece 3 of 4 · The Brain

⚙️ The Go API (Gin + SQLite)

The server that receives events, handles auth, runs the pipeline, stores everything, and answers the dashboard. It is built so tracking never blocks an HTTP response.

The endpoints

POST  /api/events        ← ingest one event or an array → returns 202 instantly
POST  /api/auth/signup   ← create account, set HttpOnly JWT cookie
POST  /api/auth/login    ← log in, set cookie
GET   /api/auth/me       ← who am I? (reads cookie)
GET   /api/stats         ← analytics for the dashboard (cookie-gated)
GET   /health            ← is the service alive? (pings DB)
GET   /metrics           ← Prometheus-style counters

The receive-and-queue trick (why it feels instant)

HTTP handler does almost nothing

It validates size, rejects junk, stamps the real IP + received-at time onto the event, drops it onto a bounded in-memory channel, and immediately replies 202 Accepted.

One background worker owns the heavy work

A single goroutine reads from that channel and runs the pipeline. Because only one writer touches the database, the write path is naturally race-free.

The "windowed" pipeline (a key design decision)

The worker collects events into a window. A window closes when it hits WINDOW_MAX_EVENTS (2048) or WINDOW_DURATION (500ms) — whichever comes first. Each closed window is one pipeline run, after which it writes (a) the cleaned events and (b) the per-stage metrics for that window. This is what makes live, sub-second metrics possible instead of only at shutdown.

Auth & cross-origin

Passwords hashed with bcrypt. Login issues an HS256 JWT in an HttpOnly cookie. CORS is locked to the exact SPA origin with credentials allowed — no wildcards. Login returns a generic 401 for both wrong email and wrong password (no user enumeration).

Back-pressure & overload

The ingest channel is bounded. If it's full, the event is dropped and counted (mable_events_dropped_total) rather than blocking the handler. The motto: freshness over completeness — tracking must not stall the UI.

Piece 4 of 4 · The Assembly Line — Deep Dive

🔧 The Pipeline Library, in depth

A generic, concurrent Pipeline[T] — an assembly line of stages. This section explains why it exists, why it's generic, how the concurrency engine works, and what every single stage does under the hood (with the real code from this repo).

1 · Why build a library at all?

The brief required the tracking API to consume a reusable pipeline, not a one-off function buried in the API. The payoff: there is exactly one piece of stream-processing code, and it has two callers:

Caller A — the API

Each ingest window builds a pipeline and runs that window's events through it on the way to SQLite. This is the production path.

Caller B — the benchmarks

The bench_test.go harness builds a near-identical pipeline to measure throughput at 10 → 1M events.

Why it matters

Because the API and the benchmark share the same code, the thing you measure is the thing you ship. No "demo pipeline vs real pipeline" drift. The library knows nothing about HTTP, SQLite, or events — it's pure stream plumbing, so it can be lifted into any other service unchanged.

2 · Why generic (Pipeline[T]) instead of the alternatives?

This was the single most important design decision in the library. Go gives several ways to build a reusable pipeline; here's why homogeneous generics won.

ApproachWhat it meansWhy it was rejected / chosen
✅ Generic Pipeline[T] One element type flows end-to-end. A stage is func(T) T-ish. Chosen. Compile-time type safety (the compiler rejects a stage that doesn't match T), no boxing of every element into an interface, and an allocation-free hot path. Reads naturally: p.Filter(...).Map(...).
❌ Type erasure (any) Channels carry interface{}; every stage type-asserts back to its concrete type. Each of millions of events would be boxed into an interface (heap allocation + GC pressure), every stage pays a runtime type assertion that can panic, and bugs surface at runtime not compile time. Fatal for a hot ingest path.
❌ Code generation Generate a bespoke pipeline per type with go generate. Extra build step, generated code to review, and worse ergonomics than real generics — which Go has had since 1.18. Solving a problem the language already solved.
❌ Hard-coded Event pipeline Skip generics; write the pipeline directly against the Mable event struct. Not reusable (the brief's core requirement), and couples the plumbing to one payload. The benchmark's TestStruct couldn't share it.
The honest trade-off we accept: a homogeneous Pipeline[T] can't change the element type mid-stream (you can't turn a RawEvent into a CleanEvent as a new type). The fix is idiomatic: make T a struct that carries its variants. That's exactly what the API does — control data like consent and server-receipt-time ride inside the event's Properties map under underscore-prefixed keys (_consent, _received_at), then get stripped before persistence. The element stays one type the whole way down.
The one place a second type appears: the Reduce[T, K, R] sink, which folds a stream of T into a keyed map[K]R — genuinely a different output type. Because Go methods can't introduce new type parameters, Reduce is a package function, not a method on Pipeline[T]. That's not a quirk; it's the type system honestly reflecting that a reduce changes the type.

3 · The concurrency engine — how an event actually moves

A pipeline is not a simple loop. When you call Collect or Reduce, the runtime (execute()) wires up a chain of goroutines connected by bounded channels of batches:

source(chan T)
   │
   ▼  [ batcher goroutine ]  groups elements into []T of up to BatchSize,
   │                         or flushes a partial batch after BatchTimeout
   ▼
head(chan []T, buffered)
   │
   ▼  [ stage 1 runner ]  ─ fans out to WorkerCount goroutines ─┐
   │                        each: read batch → Process → write   │
   ▼                                                             │ bounded
chan []T  ──▶ [ stage 2 runner: WorkerCount workers ] ──▶ chan []T  ──▶ ... 
   │                                                             │
   ▼  [ sink drain ]  runs on ONE goroutine (single-threaded accumulator)
   ▼
 SQLite / map[K]R

Why batches?

Sending one element per channel operation would make channel + scheduler overhead dominate. Grouping into []T amortises that cost. The batcher flushes either when the batch is full or a BatchTimeout (5ms) fires — so a trickle of events never stalls waiting to fill a batch.

Why bounded channels?

A buffered-but-bounded channel (ChannelBufferDepth batches) means a slow stage blocks a fast upstream stage once the buffer fills. This is back-pressure for free — memory can never balloon, no matter the input volume. That's how 1M events run in bounded RAM.

Why fan-out workers per stage?

WorkerCount goroutines (default = NumCPU()) all read from the same inbound channel, so a CPU-heavy stage uses every core. The cost: ordering is not preserved across a stage's workers — which is fine, because the only order-sensitive stage (keyed Reduce) is order-independent by construction.

Why drain the sink on ONE goroutine?

The terminal sink (the SQLite writer or the Reduce accumulator) runs single-threaded. That makes the accumulator race-free without any locks, while still enjoying all the upstream parallelism. Simple and fast.

Tunable hyper-parameters (functional options on New): WithBatchSize, WithWorkerCount, WithChannelBufferDepth, WithBatchTimeout, WithErrorPolicy, WithMetricsSink. Invalid values are clamped (e.g. WorkerCount < 1 → 1) so a degenerate config still runs correctly instead of deadlocking.

4 · Every stage type, drilled down

Each stage implements one tiny interface — that's the whole extension protocol:

type Stage[T any] interface {
    Name() string                                              // for metrics + errors
    Process(ctx, in []T) (out []T, dropped int, err error)     // batch in → batch out
}

A stage takes a batch and returns a batch (0..N out per element in), plus how many it deliberately dropped, plus an optional error. Click each to expand.

Map 1 → 1 transform

Transforms each element into a new element of the same type. Pure and trivially parallel-safe.

func (s mapStage[T]) Process(_ ctx, in []T) ([]T, int, error) {
    out := make([]T, len(in))
    for i, v := range in { out[i] = s.fn(v) }   // never drops, never errors
    return out, 0, nil
}

Why built this way

It allocates a fresh output slice rather than mutating in place, so a Map is a pure function of its input — safe to run on many workers at once with zero coordination.

Filter drop on predicate

Keeps elements where the predicate is true; everything else is counted as dropped (not an error — a drop is normal, expected load-shedding).

for _, v := range in {
    if s.pred(v) { out = append(out, v) } else { dropped++ }
}
return out, dropped, nil

Why drops are counted, not errored

Dropping a non-consenting or unknown event is the correct outcome, so it must not trip the error policy. Counting drops separately is what lets the dashboard show "X events shed at the consent gate" as a healthy signal.

Generate 1 → N fan-out

Emits zero-or-more new elements per input element. If you want the original to survive, your function returns it alongside the new ones.

for _, v := range in { out = append(out, s.fn(v)...) }   // flatten 1→N

Why it exists

Some events naturally explode into several (e.g. a "cart purchased" event → one event per line-item). Generate models that without breaking the homogeneous-type rule, since every produced element is still a T.

If branch & merge

Routes each element into one of two sub-pipelines by a predicate, runs them inline, then merges both outputs back into the single downstream stream.

thenOut, _, _ := applyStages(ctx, s.then, matched)    // run "then" stages
elseOut, _, _ := applyStages(ctx, s.els,  others)     // run "else" stages
return append(thenOut, elseOut...), drops, nil         // merge

Why sub-pipelines run inline

The branches execute synchronously within the worker that owns the batch, so each branch still benefits from the outer stage's fan-out, and an If is itself just a Stage — meaning you can nest If inside If. Sinks can't appear in a branch (a sink is terminal), only linear stages.

Deduplicate bounded-LRU drop

Drops elements whose key was seen before, using a bounded LRU seen-set (a map + doubly-linked list). This is the one built-in stage that holds mutable shared state, so it synchronises with a mutex.

s.mu.Lock(); defer s.mu.Unlock()              // all workers serialise here
for _, v := range in {
    k := s.key(v)
    if _, seen := s.seen[k]; seen { dropped++; continue }   // repeat → drop
    out = append(out, v)
    s.seen[k] = s.order.PushFront(k)
    if s.cap > 0 && s.order.Len() > s.cap {   // evict oldest when over cap
        oldest := s.order.Back(); s.order.Remove(oldest); delete(s.seen, oldest.Value)
    }
}

Why bounded (and the trade-off)

An unbounded seen-set would grow forever — a memory leak at scale. Capping it keeps memory bounded; the price is that once a key is evicted, a much-later repeat is a false negative (it slips through). capacity = 0 means exact/unbounded if you truly need it. The mutex is deliberate: exactness across the whole stream requires shared state, so dedup is a global synchronisation point by design.

Reduce terminal sink · T,K,R

Folds the entire stream into a keyed map[K]R — counts per event type, sums per user, etc. It's a sink (terminal), so it can't be chained into more T stages.

counts, metrics, _ := pipeline.Reduce(ctx, p, source,
    func(e Event) string { return e.EventType },     // key   K
    func() int { return 0 },                          // init  R
    func(acc int, _ Event) int { return acc + 1 })    // fold  (acc,T)→R

Why global, not per-batch

Analytics aggregates are only meaningful over the whole stream, so the accumulator persists across every batch. It's drained on the single sink goroutine, so the map[K]R needs no locks. And as noted above, it's a package function because it introduces the new type parameters K, R that a method on Pipeline[T] cannot.

Collect terminal sink · back-pressure

Drains the stream into a caller-provided bounded sink (this is what the API uses to write to SQLite). The ready-made ChannelSink[T] has two policies:

Block
A full sink blocks the drain, propagating back-pressure all the way upstream. Use when you must not lose data.
DropOnFull
A full sink sheds the element and counts it (returns ErrDropped). Use when freshness beats completeness.

Why two policies

Different callers have opposite needs. A billing pipeline wants Block (never drop money). A live-metrics pipeline wants DropOnFull (a stale number is worse than a missing one). The library refuses to decide for you — it exposes both and documents the consequence.

5 · Error & panic handling

Every stage call is wrapped by safeProcess, which recovers panics and converts them into a stage error (counting the whole batch as dropped) — a buggy stage can never crash a worker goroutine. Then the policy decides:

SkipAndCount
(default) drop the bad batch's output, increment the stage's error counter, keep the stream alive. Right for streaming analytics — one poison event mustn't kill ingestion.
FailFast
cancel the whole run's context on the first error so every goroutine drains and exits cleanly. Right for batch jobs where correctness > availability.

6 · Per-stage metadata (not one number)

The brief demanded per-stage metrics including errors and drops, not just one pipeline timing. Each stage emits a StageMetric: items in / out, dropped, errors, batches, total latency, p50 / p99, wall time and throughput.

The clever bit: no locks on the hot path

Each worker accumulates its own workerStat (counts + a latency slice) locally. Only when the stage finishes are the per-worker stats merged into one StageMetric and percentiles computed. So the measured path itself stays contention-free — measuring doesn't slow the thing being measured. A MetricsSink seam can forward this snapshot to ClickHouse/Prometheus.

7 · The real ingest pipeline — every stage in the API path

This is the exact assembly line the API builds for every window (from internal/ingest/pipeline.go). The order is deliberate, and below each stage is what it really does and why it sits where it sits.

① consent→ ② normalize→ ③ tracked→ ④ enrich_geo→ ⑤ churn_enrich→ ⑥ dedup→ ⑦ collect → SQLite
① Filter "consent" privacy gate, first

Drops any event that didn't carry consent: e.Properties["_consent"] == "true". The handler stamped that in-band control key.

Why first

Privacy-by-design: shed disallowed data as early as possible so no non-consenting event is ever normalised, enriched, or stored. It's also the cheapest stage, so it removes load before the expensive stages run.

② Map "normalize" clean & canonicalise

Trims whitespace, canonicalises event_type to PascalCase ("purchase" → "Purchase"), defaults a zero timestamp to now, upper-cases currency, clamps a negative amount to 0, and bounds the payload — URL ≤ 2048 chars, ≤ 64 property keys, each value ≤ 1024 chars.

Why second (before the whitelist)

This is a deliberate, correctness-driven ordering. If the whitelist ran first, "purchase" (lowercase) would be rejected as "unknown". Normalising before the tracked filter and ChurnEnrich means those stages compare against a single canonical spelling. The clamps also defend against a hostile oversized payload blowing up memory.

③ Filter "tracked" whitelist

Keeps only the seven canonical types — PageView, Click, AddToCart, Checkout, PaymentInfoAdded, Purchase, Lead — and drops everything else (counted).

Why here

Runs right after normalize so it can trust the canonical spelling. Anything unrecognised (typo, probe, or an event the product doesn't care about) is shed before the costlier enrichment stages and before it can reach the database.

④ Map "enrich_geo" add server-side context

Adds fields the browser must NOT be trusted to provide: a synthetic geo_region derived from the server-seen IP, a timezone, and device/browser parsed from the user-agent — all written into Properties, no external network calls.

Why server-side

This is the browser-vs-server split enforced in code: the script never sends IP or geo; the pipeline derives them from data only the server can see. Placed after the whitelist so we only spend enrichment effort on events we'll actually keep.

⑤ ChurnEnrich the example custom stage

The worked example of the extension protocol — a custom Stage[Event] added with .Stage(...), zero changes to the core library. It attaches a churn-probability score to Purchase events and passes everything else through untouched.

func (c ChurnEnrich) Process(_ ctx, in []Event) ([]Event, int, error) {
    for i, e := range in {
        if e.EventType == "Purchase" { e.ChurnProbability = churnScore(e) }
        out[i] = e
    }
    return out, 0, nil
}

Why it's here

It proves the library is genuinely extensible — a future engineer adds a real ML-scoring stage the same way, without touching pipeline internals. The score itself is a deterministic placeholder (a hash of the user blended with amount) standing in for a real model call.

⑥ Deduplicate "dedup" collapse replays

Key = sessionID | eventType | minute. A replayed beacon (the tracker retries on reconnect) for the same session + type within the same minute collapses to one stored event.

Why just before the sink, and the windowing caveat

It sits last so only surviving, enriched events are considered — no point de-duping things you were going to drop anyway. Because the pipeline runs in windows, the dedup LRU is per-window: a replay that straddles two windows isn't caught. Replays almost always arrive close together, and the minute in the key bounds semantic duplicates — an accepted trade-off documented in the design.

⑦ Collect → SQLite terminal sink

The surviving, cleaned events are drained into the event sink and written to SQLite in one transaction per window; the window's per-stage StageMetrics are written to stage_metrics in the same flush.

Why a sink, not another Map

Persistence is terminal — nothing flows downstream of it — and it's the boundary where the library hands off to the API's storage seam. Swapping SQLite for Postgres/ClickHouse later means changing only this sink, not the pipeline.

Recap of the "why this order": cheap privacy gate first → canonicalise so later stages can trust the data → drop unknowns before spending effort → enrich only survivors → custom scoring → de-dupe only things we'll keep → persist. Each stage assumes the guarantees established by the one before it.

Design Reasoning

Why it's built this way (in plain terms)

Tracking can't break the shop

Every tracker call is wrapped in try/catch, sends are non-blocking, and the server drops on overload instead of stalling. A broken analytics backend must never cost a sale.

Don't trust the client

IP, geo and receipt time are added server-side. Consent is checked both in the browser and again as the very first pipeline stage.

One library, two callers

The same pipeline runs inside the API and in standalone benchmarks. That keeps the "real" code path the thing that's measured and tested.

Live metrics, bounded memory

Windowed runs flush per-stage metrics every fraction of a second while keeping at most one window of events in flight — fast feedback without runaway memory.
