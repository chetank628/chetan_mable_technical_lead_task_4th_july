## Objective

This document is a complete, plain-English walkthrough of the four pieces of the Mable tracking system and how they talk to one another. It is meant for anyone who needs to understand the whole flow — frontend, backend, library, and datastore — without opening every file.

## What it covers

- The role of each piece: `ecom/`, `scripts/tracker.js`, `api/`, and `pipeline/`.
- The concrete journey of one event — a **Purchase** — from the user's click to the live dashboard.
- The contracts that tie the pieces together: HTTP endpoints, JSON shapes, Go interfaces, and the database schema.
- The integration decisions that keep tracking fast, safe, and privacy-aware.

## What it does not cover

- Deep internals of the generic pipeline library (see `plans/pipeline-working/plan.mdx`).
- Production deployment, observability dashboards, or load testing.
- Fine-grained React or Go package internals that do not affect the integration.

## Big picture

A shopper uses the **e-commerce SPA**. A tiny standalone **tracking script** watches what they do and sends well-formed events to the **Go API**. The API does not write directly to the database; it hands every event to a reusable **pipeline library** that cleans, filters, enriches, and de-duplicates it. Only the surviving output is stored, together with per-stage pipeline metrics. The SPA's dashboard then reads the stored data back via the API and shows live charts.

Each piece only knows the public contract of the next piece. The SPA does not know the pipeline exists; the tracker does not know SQLite exists; the API does not know React exists. This keeps every component replaceable.

## Concrete walkthrough: one Purchase event

Imagine a logged-in shopper with a non-empty cart clicks **Place order** on the checkout page. Here is the entire path of the `Purchase` event.

### 1. Browser: the action happens

The checkout page (`ecom/app/routes/checkout.tsx`) calls `tracker.purchase(...)` with the order id, total, currency, and items. This is a thin TypeScript wrapper over the global `window.mable` object, which is loaded from `scripts/tracker.js` via a `<script src="/tracker.js">` tag in `ecom/app/root.tsx`.

### 2. Tracker script: build the event

`tracker.js` constructs a full `IngestEvent` JSON object. It collects browser-only fields such as user agent, timezone, screen size, URL, referrer, and a session id. It coerces every `properties` value to a string to match the API's `map[string]string` contract, and promotes `amount` / `currency` to top-level fields.

<AnnotatedCode
  id="tracker-build-event"
  filename="scripts/tracker.js"
  language="js"
  code={"return {\n  event_type: type,\n  user_id: state.userId || \"\",\n  session_id: getSessionId(),\n  timestamp: new Date().toISOString(),\n  url: location.href,\n  referrer: document.referrer || \"\",\n  user_agent: ctx.user_agent || \"\",\n  amount: amount,\n  currency: currency,\n  properties: p,\n  consent: config.consent === true,\n};"}
  annotations={[
    {
      lines: "1-11",
      note: "The script deliberately does NOT send IP or geo. Those are added server-side from the connection."
    }
  ]}
/>

The event is buffered. When the batch reaches 10 events, or after 5 seconds, or when the tab hides, the script flushes the batch via `navigator.sendBeacon` (or `fetch(keepalive)`) to `POST /api/events`.

### 3. Go API: accept and queue

`api/internal/handlers/events.go` accepts a single event or a JSON array. It immediately stamps the real IP address and server receipt time onto the event, then calls `worker.Submit(...)`.

<AnnotatedCode
  id="api-ingest-handler"
  filename="api/internal/handlers/events.go"
  language="go"
  code={"ip := c.ClientIP()\nua := c.Request.UserAgent()\nnow := time.Now().UTC()\n\naccepted, dropped := 0, 0\nfor _, in := range events {\n  if h.worker.Submit(in.ToEvent(ip, ua, now)) {\n    accepted++\n  } else {\n    dropped++\n  }\n}\n\nc.JSON(http.StatusAccepted, gin.H{\n  \"accepted\": accepted,\n  \"dropped\":  dropped,\n})"}
  annotations={[
    {
      lines: "1-12",
      label: "202 Accepted",
      note: "The HTTP response returns immediately. The browser never waits for the database write."
    }
  ]}
/>

`Submit` places the event onto a bounded channel. If the channel is full, the event is dropped and counted — tracking must never block the UI.

### 4. Pipeline library: the windowed run

One background worker (`api/internal/ingest/worker.go`) drains the channel into a bounded **window**. A window closes at 2048 events or 500ms, whichever comes first. The worker then runs the entire window through one fresh `pipeline.Pipeline[Event]`.

The pipeline is built in `api/internal/ingest/pipeline.go`:

### 5. SQLite: persist events and metrics

After the window finishes, the worker writes the surviving events in one transaction and the per-stage `StageMetric` records in another. The `events` table holds the cleaned data; the `stage_metrics` table holds items in/out, drops, errors, latencies, and throughput per stage.

### 6. Dashboard: read it back

The dashboard route (`ecom/app/routes/dashboard.tsx`) calls `GET /api/stats` with credentials, which computes aggregates over the stored events and returns counts, capture times, and per-stage rollups. The dashboard renders them as charts.

## How the pipeline library works

The pipeline in `pipeline/` is a **homogeneous, concurrent, streaming pipeline**. That means every stage operates on the same element type (`Event` in the API), elements are moved in dynamically-sized batches through bounded channels, and each stage fans out across a pool of worker goroutines. The API uses it by creating a new `Pipeline[Event]` for every window of events.

### The three runtime phases

1. **Batching.** A single `batcher` goroutine reads from the source channel, accumulates elements into batches of up to `BatchSize`, and flushes any partial batch after `BatchTimeout`. It writes `[]T` batches to the first inter-stage channel.
2. **Stage execution.** Each stage launches `WorkerCount` goroutines. Each worker pulls a batch, calls `Process`, and writes the result to the next channel. Stages are chained: stage N's output channel is stage N+1's input channel.
3. **Sink drain.** A single goroutine consumes the final channel and calls the sink's `Consume` method. Because the sink is single-threaded, accumulators like `Collect` or `Reduce` need no locks.

### Back-pressure and bounded channels

Every inter-stage channel is bounded. If a downstream stage is slow, its inbound channel fills, which blocks the upstream stage's workers. That naturally throttles the whole stream without any explicit rate limiter. In the API, the worker drops events when the ingest queue is full, so the HTTP handler never blocks on a slow pipeline.

### Error policies

When a stage returns an error (or panics), the pipeline can react in two ways:

<Table
  id="pipeline-error-policies"
  columns={["Policy", "Behaviour", "When to use it"]}
  rows={[
    ["`SkipAndCount` (default)", "Drop the failing batch, increment the stage error counter, keep the pipeline running.", "Streaming analytics — one bad event must not stop ingestion."],
    ["`FailFast`", "Cancel the run context on the first error; all goroutines drain and exit.", "Batch jobs where correctness matters more than availability."],
  ]}
/>

### Per-stage metrics

Every run produces a `StageMetric` for each stage (and the sink): items in/out, dropped, errors, batch count, total latency, p50/p99 latencies, wall time, and throughput. Each worker accumulates its own stats locally and merges them only when the stage finishes, so measuring does not add contention on the hot path.

In the API, the worker writes these metrics to the `stage_metrics` table after every window, giving sub-second visibility into the pipeline without any lock on the measured path.

## Key integration decisions

### The browser-vs-server split

The browser deliberately sends only what it can know: user agent, URL, timezone, screen size, session, and user id. The server adds the IP address and receipt time, then derives geo and device fields from those. This prevents a malicious client from lying about its location or spoofing timestamps.

### The windowed pipeline

The pipeline library flushes per-stage metrics only once per run. If the API ran one perpetual pipeline, metrics would never reach the database until shutdown. The API therefore runs one bounded pipeline per window, giving sub-second metric updates while keeping at most one window of events in memory.

### Non-blocking transport at every layer

- The tracker batches events and sends them with `sendBeacon` or `fetch(keepalive)`.
- The API returns `202 Accepted` before the database write.
- The worker drops events when the queue is full rather than blocking HTTP handlers.

### Cookie-based auth

Login and signup issue an HS256 JWT in an `HttpOnly` cookie. The JavaScript never sees the token, which protects against XSS theft. Every API call uses `credentials: "include"`, and CORS is locked to the exact SPA origin.

## Verification

You can run the whole flow locally with three commands:

```bash
# 1. Start the Go API (also serves /api/events and /api/stats)
cd api && go run .

# 2. In another terminal, start the SPA
cd ecom && pnpm install && pnpm dev

# 3. Open http://localhost:5173, accept tracking, sign up, add a product to cart,
#    complete checkout, then visit http://localhost:5173/dashboard.
```

To see the pipeline directly without the browser:

```bash
cd api
# ingest a single event
curl -s -XPOST localhost:8080/api/events -H 'Content-Type: application/json' \
  -d '{"event_type":"Purchase","session_id":"s1","user_id":"u1","amount":42,"currency":"usd","consent":true}'

# read the stored metrics
curl -s localhost:8080/metrics
```
