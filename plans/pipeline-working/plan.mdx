## Objective

This plan explains the internal mechanics of the generic `pipeline` package (`/pipeline`) in `Mono`. It is a reusable, concurrent streaming pipeline library that powers both the standalone benchmark harness and the tracking API's event-processing path.

The pipeline is **homogeneous** over a single element type `T`, moves elements through **bounded channels** in dynamically-sized batches, and fans each stage out across a pool of **worker goroutines**. Back-pressure is intrinsic: a slow downstream stage naturally throttles upstream because every inter-stage channel is bounded.

### What this document covers

- The end-to-end lifecycle of a batch: source → batcher → stage runner → sink.
- The available stage types and their contracts.
- The runtime configuration levers and their defaults.
- Per-stage metrics and error handling.
- How custom stages plug into the pipeline without touching core code.

### What this document does not cover

- API design decisions or deployment topology (see `api/api-design.md` and `DESIGN.md`).
- Tracking-client event schema or the analytics DB schema.
- Performance tuning recommendations for production workloads.

## Worked example: event analytics pipeline

This example mirrors the benchmark pipeline in `bench_test.go` and is close to the tracking API ingest path. The pipeline:

1. Accepts `Event` values from a source channel.
2. Filters out event types that are not in the tracked set.
3. Enriches `Purchase` events with a churn probability score via the custom `ChurnEnrich` stage.
4. Deduplicates by `SessionID + "|" + EventType` using a bounded LRU.
5. Reduces the surviving stream into `map[string]int` — counts per event type.

### What the batcher actually does

With the default `BatchSize = 256`, the 8 events above are only one partial batch. Because the batcher also has `BatchTimeout = 5ms`, it flushes the partial batch quickly rather than waiting for 256 events. A high-volume stream would fill 256-element batches continuously, amortizing channel overhead.

### What the worker pool does

Each stage (`Filter`, `ChurnEnrich`, `Deduplicate`) runs with `runtime.NumCPU()` workers by default. Each worker independently pulls batches, calls `Process`, and writes the result. The order of events across the whole stream is **not preserved** — the duplicate `PageView` may be processed by a different worker than the original. Order-independent aggregation (count by key) is why this is safe.

### Why the sink is single-threaded

`Reduce` accumulates into a plain `map[string]int` on the goroutine that calls `p.execute`. It never needs a mutex because no other goroutine touches it; all upstream fan-out has already been serialized through the final channel. The same is true for `Collect` sinks.

### The three runtime phases

1. **Batching.** `batcher` reads elements from the source `chan T`, accumulates them into batches of up to `BatchSize`, and flushes any partial batch after `BatchTimeout`. It writes `[]T` batches to the first inter-stage channel.
2. **Stage execution.** `runStage` launches `WorkerCount` goroutines. Each worker pulls batches from the inbound channel, calls `Stage.Process` (wrapped in `safeProcess` for panic recovery), and writes the result batch to the outbound channel. Stages are chained: stage N's outbound channel is stage N+1's inbound channel.
3. **Sink drain.** `drainSink` runs on the caller's goroutine and consumes the final channel. It calls `Sink.Consume` for each batch. Because the sink is single-threaded, accumulators like `Reduce`'s `map[K]R` are race-free without locks.

The `execute` function coordinates shutdown with a cancellable `context.Context`. On `FailFast`, the first error cancels the context so all goroutines drain and exit.

The `Pipeline[T]` builder exposes fluent methods for the common primitives. Each appends a `Stage[T]` implementation to the internal `stages` slice.

    ["Deduplicate", "`.Deduplicate(name, key, capacity)`", "`key func(T) any, capacity int`", "Drops repeat keys using a bounded LRU seen-set."],
    ["Stage", "`.Stage(s)`", "`Stage[T]`", "Open extension point for any custom stage."]
  ]}
/>

### Deduplicate is a global synchronization point

The built-in `dedupStage` holds a single `sync.Mutex` and a `container/list` LRU. Every worker serializes on that mutex when processing a batch. This is intentional: exact deduplication across the whole stream requires shared state. The bounded LRU keeps memory usage predictable, but once a key is evicted a later repeat becomes a false negative.

<Callout id="dedup-tradeoff" tone="info">

Set `capacity = 0` for an exact, unbounded seen-set. Use a bounded capacity when the stream is large and approximate deduplication is acceptable.

</Callout>

## Per-stage metrics

Every run produces a `[]StageMetric` with one entry per stage plus one for the sink. `Pipeline.Metrics()` returns a copy of the most recent snapshot.

The hot path keeps worker-local stats and merges them once at the end. No locks or atomics are touched during steady-state processing.

## Error handling

`safeProcess` wraps every `Stage.Process` call in a `recover` that converts panics into counted errors. The configured `ErrorPolicy` decides what happens next.

<Callout id="error-policy" tone="warning">

- **`SkipAndCount`** (default): the offending batch's output is dropped, the stage error counter is incremented, and the pipeline keeps running. This is appropriate for streaming analytics where one bad event must not stop the whole stream.
- **`FailFast`**: the first error or panic sets `runErr`, cancels the context, and every goroutine drains. The first error is returned from the run.

</Callout>

Custom stages implement `Stage[T]` and are added with `Pipeline.Stage`. No core file needs to change. The example below is the `ChurnEnrich` stage shipped in `examples_stage.go`.
